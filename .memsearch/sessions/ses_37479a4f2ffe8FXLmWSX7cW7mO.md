1) SysML Requirements diagram concepts — core ideas and relationships

- What a SysML Requirement is
  - Core concept: a Requirement is a model element that expresses a capability or condition the system must satisfy (a “shall” clause). SysML distinguishes Functional Requirements (what the system shall do) and Non‑Functional Requirements (qualities or constraints). (Source: SysML.org requirement FAQ)
  - Evidence: SysML.org definition: “Requirement: … a capability or condition that a system must (‘shall’) satisfy. A Functional Requirement … specifies a function … whereas a Non‑Functional Requirement (NFR) specifies quality criteria …” (https://sysml.org/sysml-faq/what-is-requirement-diagram.html)

- How Use Cases are treated in SysML
  - Core concept: Use Cases are behavioral/transactional descriptions of interactions between external Actors and the System (Subject). Use Cases are often treated as high‑level functional requirements: they capture system scope and stakeholder transactions in accessible terms. SysML encourages tracing from Use Case to formal requirements when the Use Case is to be treated as a requirement. (Source: SysML.org use case FAQ)
  - Evidence: “Use Case: … represents a system transaction with an external system user, called an Actor… Use Cases are sometimes considered high‑level functional requirements. … If Use Cases are considered to be high‑level system functional requirements they should be traced to «functionalRequirement» Requirements using Refine (‘refine’) Dependencies.” (https://sysml.org/sysml-faq/what-is-use-case-diagram.html)

- Relationship between Actors, Use Cases, and Requirements
  - Semantics:
    - Actors: external entities (people, other systems, hardware) that interact with the subject system.
    - Use Cases: describe transactions (behavioral scenarios) between Actors and the Subject; they define **what** stakeholder interactions the system must support at a high level.
    - Requirements: formalize needed capabilities/constraints; Use Cases can be refined into or traced to Requirements (e.g., a Use Case “Process Payment” → one or more «functionalRequirement» entries describing required function, performance).
  - Trace pattern: Use Case —refine→ Requirement (or create Requirements from Use Case statements). This keeps stakeholder language while enabling rigorous downstream traceability.
  - Evidence: use‑case FAQ guidance on tracing Use Cases to «functionalRequirement» via refine (https://sysml.org/sysml-faq/what-is-use-case-diagram.html) and the requirement FAQ that Requirement diagrams show relationships among Requirements, elements that Satisfy them, and Test Cases that Verify them (https://sysml.org/sysml-faq/what-is-requirement-diagram.html).

- Key SysML relationships used in requirements modeling (semantics)
  - refine: links a less formal element (Use Case, stakeholder statement) to a formal Requirement that refines it.
  - satisfy: marks a model element (design or behavior) that fulfills a Requirement.
  - verify: connects a TestCase to the Requirement it verifies (test evidence).
  - deriveReqt / copy: indicate derived requirements or copies with different provenance.
  - allocate / allocation relationships (discussed below) used to map requirements to components or functions.
  - Evidence: requirement FAQ: “Requirement diagrams … show the relationships among Requirements, model elements that Satisfy them, and Test Cases that Verify them.” (https://sysml.org/sysml-faq/what-is-requirement-diagram.html)

2) Activity diagrams and functional flows — modeling and mapping

- How functional flows are modeled
  - Activity diagrams (UML/SysML Activity diagrams) model the flow of control and data between actions. They are the recommended diagrams to capture the detailed functional behavior implied by a Use Case or a Requirement.
  - An Activity represents a coordinated set of actions (or a single action) implementing some function; flows (control flows and object flows) show sequencing, decision points, concurrency (fork/join), and data passed between actions.
  - Evidence: SysML FAQ on Activity diagrams (index points to activity diagram guidance; see the Use Case FAQ advice to “Cut-over to high-level Activity diagrams ASAP!” indicating mapping from use case to activity diagrams) — see Use Case FAQ note: “Cut‑over to high‑level Activity diagrams ASAP!” (https://sysml.org/sysml-faq/what-is-use-case-diagram.html) and activity diagram FAQ (linked from sysml.org FAQ index).

- Mapping Use Case scenarios → Activity diagrams
  - Pattern:
    1. Start with the Use Case (textual or diagram): identify primary flow and alternates.
    2. For the primary flow, create an Activity with actions/stages corresponding to the Use Case steps.
    3. Model decisions (alt flows) as decision nodes/guards in the Activity; model concurrent flows with fork/join nodes.
    4. Annotate activity actions with requirements traceability (each key action can carry a «satisfy» link to the Requirement it realizes).
  - Benefit: activity diagrams encode operational detail needed to allocate behavior to design elements and to derive test cases.
  - Evidence: SysML practice guidance (SysML.org) recommends using Activity diagrams to move from high‑level use cases to implementable behavior and to tie to functional requirements (https://sysml.org/sysml-faq/what-is-use-case-diagram.html and related activity FAQ).

- Relationship between activities and requirements
  - Activities implement functional requirements: each «functionalRequirement» should be satisfied by one or more Activities or Actions (via satisfy dependencies). Conversely, activities should be traceable back to the requirement(s) they satisfy.
  - Activities can also be used to derive performance-related checks by connecting object flows to parametric constraints (for performance NFRs) and to test cases (verify).
  - Evidence: Requirement diagram purpose: “specify both Functional and Non‑Functional Requirements within the model so that they can be traced to other model elements that Satisfy them and Test Cases that Verify them.” (https://sysml.org/sysml-faq/what-is-requirement-diagram.html)

3) Allocation — allocating requirements to components, interfaces, and data flows

- How SysML expresses allocation
  - SysML provides explicit allocation relationships and allocation tables to show that a requirement or function is assigned to a block (component), activity, or other model element. The allocation keeps the “who/what is responsible” semantic visible in the model.
  - Typical allocation relationships:
    - allocate (UML/SysML allocation) / allocatedTo (tool-specific): maps behavior (Activity, Function) or Requirements → Block (component).
    - satisfy: used the other way — the component elements that implement a requirement show a «satisfy» dependency toward the Requirement.
  - Evidence: SysML specification (OMG) and FAQ describe the use of satisfy and allocation to show which model elements satisfy requirements; the Requirement FAQ: requirement diagrams show model elements that Satisfy them and Test Cases that Verify them (https://sysml.org/sysml-faq/what-is-requirement-diagram.html). OMG SysML pages (About SysML) define the language and its allocation capabilities (https://www.omg.org/spec/SysML/About-SysML).

- Defining interfaces and data flows
  - SysML uses Block Definition Diagrams (BDD) to define components (Blocks), their ports, and properties; Internal Block Diagrams (IBD) to show connectors and detailed internal structure; these diagrams define interfaces and the data flows between parts via ports and flow properties.
  - Ports / FlowProperties: define typed interfaces (signals, value flows) for data exchange.
  - Connectors: on an Internal Block Diagram, connectors bind ports/flowProperties to show actual interface wiring/data flow channels.
  - Evidence: SysML diagram taxonomy described on SysML.org (BDDs and IBDs referenced in the FAQ index; OMG Spec gives formal definitions — https://www.omg.org/spec/SysML/About-SysML).

- Traceability across layers (requirements → design → implementation → test)
  - Pattern and semantics:
    - Requirements captured in Requirement Diagrams with unique identifiers and attributes.
    - Use Cases capture stakeholder-level functional needs, refined into formal Requirements (refine).
    - Activities / actions model the required behavior derived from Use Cases / Requirements.
    - Block Definition Diagrams / Internal Block Diagrams define system structure (components, ports, interfaces).
    - Allocation relationships link behavior/requirements to structural elements (Activities → Blocks or Requirements → Blocks).
    - TestCases (a SysML model element) verify Requirements (verify). Verification artifacts can be linked to test procedures, scripts, or test results (tool‑specific).
  - Transitive traceability (V‑model): Requirement → (refine) → Design activity/behavior → (allocate) → Component → (satisfy) → Implementation element (code/hardware) → (verify) → TestCase. Keeping these relationships in the model yields a single source of truth for traceability and change impact analysis.
  - Evidence: Requirement diagram FAQ: purpose is to specify requirements within the model so they can be traced to elements that satisfy them and test cases that verify them (https://sysml.org/sysml-faq/what-is-requirement-diagram.html). OMG SysML About page provides the language intent and scope for MBSE traceability (https://www.omg.org/spec/SysML/About-SysML).

5) Practical / simplified principles to adopt (project rules for a lightweight SysML‑based approach)

- Keep requirements as first‑class model elements with stable IDs
  - Guideline: Put each requirement (functional or NFR) in Requirement diagrams with ID, short text, rationale, source, priority, verification method. This enables automated traceability and export (trace matrices).
  - Why: single source of truth, easier change management.

- Use Use Cases only for stakeholder-level scope; move quickly to Activity diagrams for implementable behavior
  - Guideline: Use Cases are useful for stakeholder communication; the Activity diagram should be the authoritative behavioral specification for engineers.

- Always connect (trace) elements with explicit SysML relationships
  - refine: Use Case → Requirement (or textual need → requirement)
  - satisfy: Design/Component → Requirement
  - verify: TestCase → Requirement
  - allocate: Activity/Function → Block
  - Why: makes left-to-right (requirements → design → test) traceability explicit and machine‑readable.

- Model interfaces and dataflows explicitly early
  - Guideline: define Block ports and FlowProperties for the most important data interfaces (credentials, signals, telemetry). Model connectors in IBDs.
  - Why: early interface definition reduces integration risk.

- Lightweight modeling discipline
  - Only create necessary diagrams for the current decision or delivery slice (to avoid model bloat).
  - Prefer smaller, well‑named actions and blocks so satisfy/refine/allocate links remain easy to manage.

- Verification planning inside the model
  - Create TestCase elements early and link them to requirements; use these to drive test plans and acceptance criteria.

- Keep traceable change logs
  - When a requirement changes, update the Requirement element and rely on model queries (or allocation tables) to enumerate impacted Use Cases, Activities, Blocks, and TestCases.

- Tool pragmatics for a simplified MBSE practice
  - If you are not using a heavy SysML toolchain, adopt a disciplined mapping in whatever tool you use (e.g., lightweight modeling in a wiki or spreadsheet) that preserves the same relationships (Requirement ID, Satisfier, Verification).
  - If you have a SysML tool, exploit its traceability queries and allocation matrices.

6) Short checklist for implementing a simple SysML‑based requirements workflow
- Step 1: Capture all stakeholder statements and assign unique requirement IDs.
- Step 2: Create a top‑level Use Case diagram for scope; mark Use Cases that will be formal Requirements.
- Step 3: For each high‑priority Use Case, author an Activity diagram that implements the primary flow(s).
- Step 4: Create Blocks (components) and IBDs for interfaces needed to implement the Activities.
- Step 5: Add explicit relationships: refine (use case → requirement), satisfy (block or activity → requirement), allocate (activity → block), verify (test → requirement).
- Step 6: Maintain a Requirement→Test trace table and run impact analysis when requirements change.
