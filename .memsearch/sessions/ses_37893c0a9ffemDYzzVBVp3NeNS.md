Best practices (concise) for using Chalk in user-facing CLI error/warning output and for avoiding noisy stack traces.
Each claim includes source evidence (GitHub permalink) and a short explanation.

1) Define theme helpers (error / warning) and use them consistently
**Claim**: Create small, reusable style constants (e.g. error, warning) and use them for all user-facing messages.
**Evidence** (Chalk README):
https://github.com/chalk/chalk/blob/aa06bb5ac3f14df9fda8cfb54274dfc165ddfdef/readme.md#L95-L99
```js
const error = chalk.bold.red;
const warning = chalk.hex('#FFA500'); // Orange color

console.log(error('Error!'));
console.log(warning('Warning!'));
```
**Explanation**: A single place for styles keeps visual language consistent, makes it easy to change color/weight, and separates styling from message text.

2) Use a stderr-configured Chalk instance when printing errors
**Claim**: Use Chalk’s stderr instance (chalkStderr) for messages printed to stderr so color-detection is correct for that stream.
**Evidence** (Chalk README):
https://github.com/chalk/chalk/blob/aa06bb5ac3f14df9fda8cfb54274dfc165ddfdef/readme.md#L151-L154
```text
chalkStderr contains a separate instance configured with color support detected for stderr stream instead of stdout.
```
**Explanation**: Some terminals/CI treat stdout/stderr differently for color support; using chalkStderr ensures the colors you add to error output are only used when stderr supports them.

3) Respect terminal color settings (NO_COLOR / FORCE_COLOR / flags)
**Claim**: Honor environment/flags for color support instead of forcing color unconditionally.
**Evidence** (Chalk README):
https://github.com/chalk/chalk/blob/aa06bb5ac3f14df9fda8cfb54274dfc165ddfdef/readme.md#L143-L149
```text
Can be overridden by the user with the flags `--color` and `--no-color`...
use the environment variable `FORCE_COLOR=1` ... `FORCE_COLOR=0` to forcefully disable.
```
**Explanation**: Respecting user/CI preferences avoids emitting stray ANSI codes in logs and makes your CLI composable in scripts and CI.

4) Surface short, actionable error messages; show stack only behind a debug/trace flag
**Claim**: Print a concise, human-friendly error plus a hint; show full stack only when the user enables a trace/verbose option.
**Evidence** (Commander example using a --trace flag to conditionally print debug/trace info):
https://github.com/tj/commander.js/blob/8247364da749736570161e95682b07fc2d72497b/examples/hook.js#L22-L34
```js
program
  .option('--trace', 'display trace statements for commands')
  .hook('preAction', (thisCommand, actionCommand) => {
    if (thisCommand.opts().trace) {
      console.log('>>>>');
      console.log(`About to call action handler for subcommand: ${actionCommand.name()}`);
      console.log('arguments: %O', actionCommand.args);
      console.log('options: %o', actionCommand.opts());
      console.log('<<<<');
    }
  });
```
**Explanation**: Users care about the error and next steps; stacks are useful to developers. Gate verbose stacks behind an explicit flag so normal users aren’t overwhelmed.

5) Centralize error formatting (single place to colorize / shorten messages)
**Claim**: Centralize error formatting (color, message templates, whether to include stack) so all UX changes are local.
**Evidence** (Commander: configureOutput example showing central formatting hooks):
https://github.com/tj/commander.js/blob/8247364da749736570161e95682b07fc2d72497b/examples/configure-output.js#L9-L15
```js
program.configureOutput({
  writeOut: (str) => process.stdout.write(`[OUT] ${str}`),
  writeErr: (str) => process.stdout.write(`[ERR] ${str}`),
  // Highlight errors in color.
  outputError: (str, write) => write(errorColor(str))
});
```
**Explanation**: Centralizing output formatting makes it trivial to (a) swap colors/themes, (b) add structured output modes, and (c) ensure errors are consistently concise.

6) Prefer “helpful short message + hint” over printing full help/stack; use built-in helpers for friendly errors
**Claim**: Use the CLI framework’s “show help after error” or suggestion helpers to produce short, actionable messages (instead of long stacks).
**Evidence** (Commander README: showHelpAfterError / showSuggestionAfterError):
https://github.com/tj/commander.js/blob/8247364da749736570161e95682b07fc2d72497b/Readme.md#L835-L857
```js
program.showHelpAfterError();
// or
program.showHelpAfterError('(add --help for additional information)');

program.showSuggestionAfterError(false);
```
**Explanation**: Short messages with a hint (e.g., “(add --help)”) solve 80% of user problems without dumping internal traces. Reserve long form outputs for debug flows.

7) Offer a machine-output mode (JSON) and keep human output separate
**Claim**: Provide a --json or similar flag that emits structured output on stdout; send human-friendly colored errors to stderr.
**Evidence** (Commander configureOutput shows how to override writes — reuse to implement JSON vs human output):
https://github.com/tj/commander.js/blob/8247364da749736570161e95682b07fc2d72497b/examples/configure-output.js#L9-L12
```js
program.configureOutput({
  writeOut: (str) => process.stdout.write(`[OUT] ${str}`),
  writeErr: (str) => process.stdout.write(`[ERR] ${str}`),
});
```
**Explanation**: Structured output is easy to parse in automation and should be uncolored JSON on stdout; human errors (colored, abbreviated) belong on stderr. Centralized write functions make this easy.
